# 消息队列堆积告警处理方案

## 告警名称
- **告警名**: `MessageQueueBacklog`
- **告警级别**: 严重
- **触发条件**: 消息队列中未处理消息数持续5分钟超过10000条，或处理延迟超过1小时

## 问题描述
当消息队列中消息堆积无法及时处理时，会导致：
- 实时业务处理延迟，影响用户体验
- 消息处理成功率下降
- 队列持续增长导致存储紧张
- 系统可能级联故障

## 排查步骤

### 步骤1: 获取当前时间
**工具**: `get_current_time`
**目的**: 确定堆积开始的时间

### 步骤2: 查询消息队列监控指标
**工具**: `query_metrics`
**参数要求**:
- **指标**: `queue_depth, consumer_lag, message_rate, processing_rate`
- **时间范围**: 最近1小时
- **队列名称**: `all` (查询所有队列)

**查询目的**: 了解队列堆积趋势和消费速率变化

### 步骤3: 查询应用日志
**工具**: `query_logs`
**参数要求**:
- **地域**: `ap-guangzhou`
- **日志主题**: `application-logs`
- **时间范围**: 最近30分钟
- **查询条件**: `queue OR consumer OR processing_failed OR exception`

**查询示例**:
```
地域: ap-guangzhou
日志主题: application-logs
时间范围: 2024-01-20 14:00:00 到 2024-01-20 14:30:00
查询语句: "message queue" OR "consumer" OR "failed" OR "exception"
```

### 步骤4: 检查消费者状态
查看消息消费者是否正常运行：
- 消费者实例数
- 消费者是否在线
- 消费者处理速率
- 是否有消费者崩溃重启

### 步骤5: 分析单条消息处理时间
检查应用日志中的处理记录：
- 单条消息平均处理时间
- 是否有异常耗时的消息
- 处理失败并重试的情况

## 常见原因分析

### 原因1: 消费者处理能力不足
**特征**:
- 消息生产速率 > 消费速率
- 消费者处理单条消息耗时较长
- 消费者实例数量没有对应增加
- 消息队列深度持续上升

**处理方案**:
1. **立即扩容**: 增加消费者实例数量
2. **优化处理逻辑**: 检查消息处理代码，看是否有性能瓶颈
3. **并发调优**: 增加消费者线程数或批处理大小
4. **缓存优化**: 引入缓存层减少后端系统调用
5. **监控**: 持续监控消费速率变化

### 原因2: 消费者异常或崩溃
**特征**:
- 消费者实例数量减少
- 应用日志中有异常堆栈
- 特定时间点消费速率突然下降至0
- 消息处理失败率突然升高

**处理方案**:
1. **查看崩溃日志**: 检查消费者的日志找出崩溃原因
2. **重启消费者**: 重启异常的消费者实例
3. **修复代码**: 如果是代码bug导致崩溃，发布补丁
4. **增加监控**: 添加崩溃告警，快速发现问题
5. **自动恢复**: 配置容器或进程监控，自动重启崩溃的消费者

### 原因3: 后端系统响应缓慢
**特征**:
- 消费者在处理消息时大量时间用于等待后端系统
- 后端数据库或api响应延迟高
- 消费者线程被阻塞
- 应用日志中有超时错误

**处理方案**:
1. **诊断后端系统**: 检查数据库、下游API是否有问题
2. **增加超时时间**: 如果是网络波动，适度增加超时时间
3. **启用重试**: 配置失败消息的重试机制
4. **降级处理**: 对于非关键业务，缺少后端数据时使用默认值
5. **隔离**: 为不同优先级消息使用不同队列和消费者

### 原因4: 消息处理出错导致重试
**特征**:
- 消息处理失败率高
- 同一条消息重复出现在日志中
- 失败消息被多次重试
- 重试队列堆积

**处理方案**:
1. **分析错误原因**: 查看处理失败的错误信息
2. **修复数据问题**: 如果是数据格式或内容问题，修正源头
3. **优化重试策略**: 配置指数退避重试
4. **死信队列**: 设置死信队列，避免无限重试
5. **告警通知**: 配置处理失败告警，及时发现问题

### 原因5: 流量突增
**特征**:
- 消息生产速率突然增加
- 与业务活动相关（营销活动、大促等）
- 短时间内队列快速堆积
- 消费速率已经优化到极限

**处理方案**:
1. **提前扩容**: 预知有流量高峰时提前增加消费者
2. **流量控制**: 在生产端实现限流，防止队列溢出
3. **优先级处理**: 为不同优先级消息设置不同的消费策略
4. **队列分片**: 使用多个队列分散流量
5. **容量规划**: 根据业务预测提前规划消费能力

## 紧急处理措施

### 立即操作（5分钟内）
1. **增加消费者**: 立即扩容消费者实例，提高消费速率
2. **确认消费者状态**: 确保所有消费者都在正常运行
3. **检查后端系统**: 确保消息处理依赖的系统正常

### 短期措施（30分钟内）
1. **分析日志**: 确定消息堆积的根本原因
2. **调优参数**: 如果是处理缓慢，增加批处理大小或线程数
3. **临时限流**: 在生产端限流，防止堆积继续增长
4. **监控**: 实时监控消费速率和队列深度

### 长期优化
1. **性能优化**: 优化消息处理逻辑，提高单位时间处理能力
2. **架构优化**: 考虑消息分片、多队列、优先级队列等方案
3. **监控完善**: 添加消费者健康检查和自动恢复机制
4. **容量规划**: 根据业务增长规划消费能力

## 验证步骤
1. 确认队列深度开始下降
2. 验证消费速率 > 生产速率
3. 观察消费者日志无异常
4. 持续监控30分钟确保队列最终清空

## 预防措施
1. **合理配置**: 根据流量预测配置消费者及并发数
2. **监控告警**: 设置队列深度70%为预警，90%为严重
3. **定期演练**: 定期进行消息队列故障演练
4. **自动扩容**: 配置基于队列深度的自动扩缩容

## 相关告警
- `HighCPUUsage`: 消费者CPU过高
- `HighMemoryUsage`: 消费者内存过高
- `SlowResponse`: 消息处理响应缓慢
