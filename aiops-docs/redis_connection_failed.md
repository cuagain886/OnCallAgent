# Redis连接失败告警处理方案

## 告警名称
- **告警名**: `RedisConnectionFailed`
- **告警级别**: 严重
- **触发条件**: Redis连接失败或连接断开持续2分钟

## 问题描述
当应用无法连接到Redis时，会导致：
- 缓存不可用，所有请求查询后端存储
- 系统吞吐能力下降
- 后端数据库压力升高
- 如果依赖Redis做会话存储，可能导致会话丢失

## 排查步骤

### 步骤1: 获取当前时间
**工具**: `get_current_time`
**目的**: 确定连接失败的具体时间

### 步骤2: 查询应用连接日志
**工具**: `query_logs`
**参数要求**:
- **地域**: `ap-guangzhou`
- **日志主题**: `application-logs`
- **时间范围**: 最近15分钟
- **查询条件**: `redis AND connection OR Redis exception OR timeout`

**查询示例**:
```
地域: ap-guangzhou
日志主题: application-logs
时间范围: 2024-01-20 14:30:00 到 2024-01-20 14:45:00
查询语句: "redis" AND ("connection refused" OR "timeout" OR "exception")
```

### 步骤3: 检查Redis服务状态
查看应用日志中的最后连接记录：
- 连接失败的原因（timeout、connection refused等）
- 最后一次成功连接的时间
- 连接池状态
- 重试次数和间隔

### 步骤4: 查询系统资源指标
**工具**: `query_metrics`
**参数要求**:
- **指标**: `redis_server_status, redis_memory, redis_connected_clients, network_latency`
- **时间范围**: 最近30分钟

**查询目的**: 了解Redis服务器的健康状态

### 步骤5: 检查网络连通性
查看应用到Redis的网络连接：
- Redis服务器IP和端口
- 是否能ping通Redis服务器
- 防火墙规则是否允许
- 应用所在网络是否有问题

## 常见原因分析

### 原因1: Redis服务崩溃或停止
**特征**:
- 连接立即refused而不是timeout
- Redis所在服务器的CPU和内存正常
- 所有应用都无法连接
- Redis进程不存在

**处理方案**:
1. **确认Redis状态**: 检查Redis进程是否正在运行
2. **查看Redis日志**: 分析Redis为什么停止
3. **启动Redis**: 重启Redis服务
4. **数据验证**: 启动后验证数据完整性
5. **根因分析**: 分析导致crash的原因

### 原因2: Redis内存溢出或达到上限
**特征**:
- Redis占用内存接近100%
- Redis开始拒绝写入请求
- OOM kill消息
- 连接可以建立但get/set操作失败

**处理方案**:
1. **增加内存**: 增加Redis可用内存上限
2. **清理数据**: 检查是否有异常占用内存的数据
3. **优化策略**: 设置合理的過期策略（TTL）
4. **eviction策略**: 配置内存达到上限时的淘汰策略
5. **监控**: 监控Redis内存使用趋势

### 原因3: 网络连接问题
**特征**:
- 连接timeout而非refused
- Redis服务正常运行
- 间断性连接失败
- 本地ping正常但redis连接不上

**处理方案**:
1. **检查防火墙**: 确保防火墙规则允许访问Redis端口
2. **检查网络**: 排查应用与Redis间的网络问题
3. **DNS解析**: 如使用域名，验证DNS解析正确
4. **所有者确认**: 确认是否是VPC/安全组配置问题
5. **增加超时时间**: 临时增加连接超时时间

### 原因4: Redis配置参数错误或连接池配置不合理
**特征**:
- Redis正常运行但应用连接失败
- 错误信息中有"invalid password"或"wrong password"
- 本地测试能连接但应用程序不能
- Redis配置中禁用了TCP连接

**处理方案**:
1. **验证连接参数**: 检查host、port、password等配置
2. **测试连接**:  使用redis-cli尝试本地连接
3. **应用配置**: 更正应用中的Redis连接配置
4. **连接池参数**: 调整min/max connections、timeout等参数
5. **重启应用**: 应用配置修改后重启生效

### 原因5: Redis版本升级或配置变更
**特征**:
- 连接失败与Redis升级或配置变更时间相符
- 新版本Redis API有变化
- 某些命令不支持或参数改变
- 连接到了新的Redis实例

**处理方案**:
1. **确认变更**: 确认是否有最近的Redis升级或迁移
2. **查看日志**: 检查Redis变更日志
3. **版本兼容性**: 检查应用与新Redis版本的兼容性
4. **升级驱动**: 必要时升级Redis Java驱动库
5. **测试**: 在测试环境充分验证后再迁移生产

### 原因6: Redis连接数达到上限
**特征**:
- 部分连接成功，部分连接失败
- Redis的connected_clients接近maxclients上限
- 并发请求较大时容易失败
- Redis日志中有关于连接拒绝的信息

**处理方案**:
1. **增加连接限制**: 提高Redis的maxclients参数
2. **连接池优化**: 优化应用侧连接池的min/max配置
3. **连接复用**: 确保应用正确复用连接而非每次新建
4. **扩容**: 增加Redis实例或使用集群
5. **监控**: 监控连接数使用情况

## 紧急处理措施

### 立即操作（5分钟内）
1. **确认Redis状态**: 检查Redis进程是否运行，telnet测试连通性
2. **重启Redis**: 如果进程停止，重启Redis服务
3. **启用缓存降级**: 配置应用不依赖Redis，直接查询后端

### 短期措施（30分钟内）
1. **分析连接日志**: 确定连接失败的具体原因
2. **修正配置**: 如是配置问题，更正后重启应用
3. **扩容**: 如是负载问题，增加Redis容量或实例
4. **通知团队**: 通知应用团队和基础设施团队

### 长期优化
1. **主从/集群**: 部署Redis主从或集群提高可用性
2. **监控完善**: 添加Redis连接、内存、响应时间等监控
3. **自动恢复**: 配置自动故障转移和重连机制
4. **容量规划**: 根据业务增长规划Redis容量

## 验证步骤
1. 确认Redis连接恢复正常
2. 验证缓存能正常读写
3. 检查应用日志无连接异常
4. 观察缓存命中率恢复到正常水平

## 预防措施
1. **连接监控**: 监控Redis连接数和连接失败率
2. **定期备份**: 定期备份Redis数据
3. **多副本**: 部署Redis主从或集群提高可用性
4. **性能测试**: 定期进行Redis性能和容量压测

## 相关告警
- `SlowResponse`: 响应时间过长（可能因缺少缓存）
- `HighCPUUsage`: CPU使用率过高
- `HighMemoryUsage`: 内存使用率过高
- `DatabaseConnectionPoolExhausted`: 数据库连接池耗尽（缓存不可用级联）
