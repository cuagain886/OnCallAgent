# 数据库连接池耗尽告警处理方案

## 告警名称
- **告警名**: `DatabaseConnectionPoolExhausted`
- **告警级别**: 严重
- **触发条件**: 数据库连接池可用连接数为0，持续2分钟

## 问题描述
当数据库连接池中的所有连接都被占用且无法释放时，会导致：
- 新的数据库请求无法执行，队列积压
- 应用响应变慢甚至超时
- 其他服务可能级联超时
- 如果长时间持续，可能导致应用崩溃

## 排查步骤

### 步骤1: 获取当前时间
**工具**: `get_current_time`
**目的**: 确定告警发生的具体时间范围

### 步骤2: 查询application日志
**工具**: `query_logs`
**参数要求**:
- **地域**: `ap-guangzhou`
- **日志主题**: `application-logs`
- **时间范围**: 最近15分钟
- **查询条件**: `connection AND pool OR ConnectionPoolException OR timeout`

**查询示例**:
```
地域: ap-guangzhou
日志主题: application-logs
时间范围: 2024-01-20 14:30:00 到 2024-01-20 14:45:00
查询语句: "Unable to get a connection" OR "Connection pool is exhausted"
```

### 步骤3: 查询数据库慢查询日志
**工具**: `query_logs`
**参数要求**:
- **地域**: `ap-guangzhou`
- **日志主题**: `database-slow-query`
- **查询条件**: `execution_time:>5000`

**查询目的**: 找出是否有长时间运行的SQL导致连接无法释放

### 步骤4: 分析正在执行的SQL
查看应用日志中的SQL执行记录：
- SQL语句内容
- 执行耗时
- 涉及的表名
- 是否有全表扫描

### 步骤5: 查询系统资源
**工具**: `query_metrics`
**参数要求**:
- **指标**: `database_connections_active, database_connections_idle, network_io`
- **时间范围**: 最近30分钟

## 常见原因分析

### 原因1: 长事务或锁等待
**特征**:
- 连接被长期占用但无法释放
- 数据库中有锁等待
- 应用日志中有SQL执行但长时间未返回
- 活跃连接数长时间不变

**处理方案**:
1. **立即操作**: 查看数据库中长时间运行的事务
2. **Killing Session**: 确认无损失后，kill掉长时间持有连接的session
3. **应用重启**: 如果应用持有大量连接未释放，考虑重启应用
4. **优化SQL**: 分析并优化执行缓慢的SQL语句
5. **调整超时**: 设置连接最大生命周期，防止僵尸连接

### 原因2: 连接池配置过小
**特征**:
- 启用连接池但配置的连接数太少
- 流量正常但连接池刚好满载
- 没有连接泄漏，但申请连接的队列很长
- 与流量高峰同步出现

**处理方案**:
1. **临时方案**: 增加连接池大小配置（min/max connections）
2. **重启应用**: 使用新配置重启应用
3. **监控**: 监控高峰期连接使用情况
4. **容量规划**: 根据业务规模合理设置连接池大小
5. **长期优化**: 优化SQL执行效率，减少连接占用时间

### 原因3: 连接泄漏
**特征**:
- 活跃连接数逐渐增加但不下降
- 应用代码中获取连接后未正确释放
- 连接被异常中断但在连接池中仍资源占用
- 启用连接池但应用长时间运行后问题越来越严重

**处理方案**:
1. **识别泄漏**: 检查应用代码中所有的数据库连接获取点
2. **检查finally块**: 确保所有连接在finally中或try-with-resources中关闭
3. **监控连接生命周期**: 添加日志记录连接的分配和释放
4. **设置连接超时**: 设置 `idleTimeout` 自动关闭长时间未使用的连接
5. **自动回收**: 启用连接池的验证查询功能，定期检测和回收死连接

### 原因4: 并发请求突增
**特征**:
- 流量突然增加
- 短时间内连接使用率从低变满
- 释放速度跟不上申请速度
- 日志中有大量的"等待连接"信息

**处理方案**:
1. **立即扩容**: 增加应用实例数量以分散连接需求
2. **启用限流**: 限制并发请求数量
3. **优化SQL**: 检查是否有N+1查询问题
4. **使用缓存**: 减少数据库查询次数
5. **数据库层面**: 检查数据库端连接是否也接近上限

### 原因5: 应用代码异常导致连接无法释放
**特征**:
- 特定功能模块出现问题时触发
- 连接一旦使用就无法释放
- 异常堆栈中没有数据库相关错误
- 重启应用后问题消失

**处理方案**:
1. **分析堆栈**: 查看最近部署的代码变更
2. **回滚测试**: 回滚到上一个稳定版本验证
3. **代码审查**: 重点审查数据库连接相关代码
4. **单元测试**: 补充单测覆盖异常场景
5. **规范使用**: 使用Spring Data JPA或ORM框架，自动管理连接生命期

## 紧急处理措施

### 立即操作（5分钟内）
1. **重启应用**: 快速恢复连接池的可用连接
2. **确认数据库**: 检查数据库端是否有长事务，必要时kill session
3. **启用限流**: 限制新连接请求，防止队列继续增长

### 短期措施（30分钟内）
1. **分析日志**: 确定连接耗尽的根本原因
2. **调整参数**: 如果是配置问题，增加连接池大小
3. **应用隔离**: 考虑为不同模块使用独立的连接池
4. **扩容**: 增加应用实例数量分散连接压力

### 长期优化
1. **代码审查**: 确保所有数据库操作都正确处理连接释放
2. **性能优化**: 优化SQL和缓存策略，减少连接占用时间
3. **监控完善**: 添加更细粒度的连接池监控指标
4. **压力测试**: 定期进行连接池压力测试，找出临界阈值

## 验证步骤
1. 确认可用连接数不为0
2. 检查应用日志无"连接池耗尽"相关错误
3. 验证新的数据库请求能正常执行
4. 持续监控连接池使用情况30分钟

## 预防措施
1. **合理配置**: 根据应用并发数合理设置连接池大小
2. **连接验证**: 启用连接池的test-on-borrow或test-on-return功能
3. **生命周期管理**: 定期关闭长时间空闲或异常的连接
4. **监控告警**: 设置连接池使用率70%为预警

## 相关告警
- `SlowResponse`: 响应时间过长
- `HighCPUUsage`: CPU使用率过高（可能是因为线程等待）
- `DatabaseSlowQuery`: 数据库慢查询
